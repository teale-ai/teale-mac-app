## Chat app foundation — all four phases of the Claude-style chat app

**Branch:** `taylorhou/chat-app`

### Goal
User wants a consumer-focused chat experience: 1:1 DMs + group chats on both macOS and iPhone, with an AI orchestrator present in every conversation from creation. Unsloth was floated as an "AI harness" to borrow from; researched and concluded it's Python + TS/Vite (Unsloth Studio) — wrong stack, treated as feature-checklist inspiration only.

### Key architectural decision
**Unified Mac binary** stays as-is: `InferencePoolApp` = chat UI + inference supply node + MenuBarExtra in one install. Splitting supply/demand apps would shrink the supply pool and raise WAN relay costs, which directly conflicts with Teale's cost-minimization goal. iPhone stays demand-only (iOS thermals/battery unsuitable as supply). Memory at `project_supply_demand.md` was updated to reverse the prior "Mac is headless" direction.

### Phase 1 — Unified ChatService storage + macOS sidebar (MVP)

Made `ChatKit.ChatService` the single source of truth on both platforms:

- **Deleted** `Sources/AppCore/Persistence/ConversationStore.swift` + `ConversationModels.swift` (old macOS JSON-per-file store)
- **Deleted** `Sources/TealeCompanion/Models/CompanionConversation.swift` (iOS in-memory store)
- **Added** `ChatKit` to InferencePoolApp's `Package.swift` dependencies
- **AppState.swift** — added `chatService: ChatService` + `currentUserID: UUID` (persisted as `teale.chatUserID` in UserDefaults); wired `aiParticipant.onInferenceRequest` to route through `engine.generate`; `initializeAsync` now loads conversations and seeds a default DM with `AgentConfig(autoRespond: true, mentionOnly: false)`
- **CompanionAppState.swift** — same pattern (persistent local UUID, non-optional ChatService, default DM seeded)
- **Rewrote `InferencePoolApp/Views/ChatView.swift`** — split into `ConversationListSidebar` + `ConversationDetailView` with HSplitView; sidebar has "New Chat" / "New Group" menu; detail view uses `chatService.activeMessages` + `aiParticipant.isGenerating` streaming; extracted `ModelPickerButton`
- **Rewrote `TealeCompanion/Views/CompanionChatView.swift`** — reads from `chatService.activeMessages` (`DecryptedMessage`), streams via `aiParticipant.streamingText`; new-chat button calls `createDM`
- **ChatService.swift** — extended `createDM` and `createGroup` to accept `agentConfig:` + `title:` params so the caller can set `autoRespond`/`mentionOnly` per-conversation
- Fixed `ConversationListView.swift`, `CompanionSettingsView.swift`, `TealeCompanionApp.swift` to use non-optional chatService

### Phase 2 — Orchestrator tool loop in AIParticipant

Replaced `AIParticipant.generateResponse` with `runTurn(conversation:chatService:participants:)` that iterates up to 4 times:
- Build request with tool schemas appended to the system prompt
- Stream inference; parse `<tool_call>{"tool":"…","params":{…}}</tool_call>` out of the response via `ToolCallParser`
- On tool call: persist `.toolCall` message → run via `ToolRegistry` → persist `.toolResult` message → re-enter inference with the updated history
- On plain text: persist as `.aiResponse` and exit

**New files**
- `Sources/ChatKit/Services/ToolRegistry.swift` — `ToolSchema`, `ToolCall`, `ToolOutcome`, `ToolHandler` protocol, `@MainActor` `ToolRegistry`, `ToolCallParser.extract(from:)`
- `Sources/ChatKit/Services/CalendarToolHandler.swift` — EventKit-backed `calendar_read` with `NSCalendarsFullAccessUsageDescription` permission handling

**ChatService** — added `insertToolCall(_:, conversationID:)` and `insertToolResult(_:, conversationID:)` helpers that route through the existing encryption + broadcast path (JSON-encoded content).

**AIParticipant.buildRequest** — tool-call messages are fed back as role="assistant" wrapped in `<tool_call>…</tool_call>`; tool-result messages as role="user" wrapped in `<tool_result>…</tool_result>` — keeps assistant/observation alternation clean.

**Views** — `.toolCall` and `.toolResult` get distinct bubbles (macOS: wrench icon + disclosure group for params, checkmark/warning icon + disclosure for results. iOS: compact rows).

Build: clean build 53.94s, no errors.

### Phase 4 — Sub-agent dispatch tool + code-block artifacts

**New files**
- `Sources/ChatKit/Services/SubAgentDispatchHandler.swift` — `dispatch_specialist(role, task)` tool. Re-enters inference with a narrow system prompt ("You are a {role}…") via a captured `inferenceStream` closure; bounded by the orchestrator's iteration cap
- `Sources/ChatKit/Services/MessageContentSegmenter.swift` — splits text by triple-backtick fences into `.text`/`.code(language:code:)` segments; unterminated fences gracefully degrade to plain text

**Registration** — both `AppState` and `CompanionAppState` now register both tools with a shared `inferenceStream` closure:
```
toolRegistry.register(CalendarToolHandler())
toolRegistry.register(SubAgentDispatchHandler(inferenceStream: inferenceStream))
chatService.aiParticipant.toolRegistry = toolRegistry
chatService.aiParticipant.onInferenceRequest = inferenceStream
```

**Code-block rendering**
- macOS `TextBubble` iterates the segmenter output; code segments render inside `CodeBlockView` with a monospaced font, rounded bordered background, optional language tag row, and a copy-to-clipboard button
- iOS `CompanionTextBubble` inlines code segments as monospaced blocks within the bubble (no copy button on mobile)

**Table rendering**: explicitly deferred — markdown table parsing is nontrivial and wasn't on the critical path for the Claude-app feel the user asked for.

Build: clean build 52.95s, no errors.

### Final state

Every conversation — DM or group — has the AI orchestrator present from creation. The orchestrator can:
1. Call calendar tools (EventKit read) and surface results as persisted, encrypted `.toolResult` messages
2. Dispatch focused sub-tasks to specialist personas via `dispatch_specialist`
3. Render code blocks as distinct monospaced artifacts in text bubbles

Both the Mac and iPhone apps read from the same `ChatKit.ChatService` store, with file-compatible `conversations.json` encoding so a conversation created on one platform renders on the other.

macOS ships as one unified binary (chat UI + inference supply node); iPhone stays demand-only; the "Contribute compute" toggle lets Mac users opt out of the supply side without giving up the chat UI.

## Mesh-LLM backend — mirror teale-node PR #10 in the Mac app

**Branch:** `taylorhou/mesh-llm-mac` (renamed from `taylorhou/london`)

### Goal
Port the `mesh` inference backend from teale-node (PR teale-ai/teale-node#10) so the Mac app can target a Mesh-LLM cluster as its inference source, alongside `localMLX` / `llamaCpp` / `exo`. Mesh-LLM is a distributed llama.cpp deployment exposing an OpenAI-compatible HTTP API at `http://localhost:9337/v1` with a web console at `:3131`.

### Key wire-level difference
Mesh-LLM has **no `/health` endpoint**. Readiness is probed via `GET /v1/models` instead. Everything else (SSE streaming of `POST /v1/chat/completions`) is identical to `llama-server`.

### Changes landed
- **`Sources/MeshLLMKit/MeshLLMProvider.swift`** — new actor conforming to `InferenceProvider`. Attach-by-default; spawns `mesh-llm serve <serveArgs>` only when `binary` is set. Readiness polls `/v1/models` with 120s timeout at 1s intervals; auto-discovers the model id from `data[0].id` when `modelID == nil`. SSE streaming duplicated from `ExoProvider` pattern. `MeshLLMProviderError` covers readiness timeout, empty model list, subprocess death, spawn failure, and server errors.
- **`Package.swift`** — new `MeshLLMKit` target; added as dep of `AppCore` and `InferencePoolApp`.
- **`Sources/AppCore/AppState.swift`** — `InferenceBackend.meshLLM = "mesh_llm"`, UserDefaults keys (`teale.meshllm_{endpoint,port,model_id,binary_path,serve_args}`), observable props (`meshLLMEndpoint`, `meshLLMPort`, `meshLLMModelID`, `meshLLMBinaryPath`, `meshLLMServeArgs`, `meshLLMStatusMessage`, `meshLLMDiscoveredModelID`), provider init, and switches in `currentBaseProvider` / `applyInferenceBackendSelection` / `inferenceEngineName` / `displayName`. Added `refreshMeshLLMStatus()` that surfaces readiness + resolved model to the Settings UI.
- **`Sources/InferencePoolApp/Views/SettingsView.swift`** — conditional block for `.meshLLM` with Endpoint / Port / Model ID / Binary Path / Serve Args fields, status line with readiness indicator, resolved-model display, "Probe Mesh-LLM" button, `Link` to `http://localhost:3131`, and install-hint text.

### Design notes
- SSE parsing logic now lives in three providers (LlamaCpp, Exo, MeshLLM). Filed as a TODO in `MeshLLMProvider.swift`: extract a shared `OpenAIStreamClient` with a parameterized `readinessPath` and migrate all three. Deferred to keep this PR focused.
- No changes to chat UI / model picker — `engine.generate()` already abstracts the active provider, so switching backend via Settings transparently flows through to chat.
- No model-download UI for mesh-llm (users run `mesh-llm download <name>` in their CLI; spelled out in the install-hint).

### Build verification
`swift build` → clean (96.95s cold, 3.40s incremental). No new warnings/errors introduced; pre-existing Swift 6 sendable-capture warnings in unrelated code remain.
