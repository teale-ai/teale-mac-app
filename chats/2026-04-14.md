# 2026-04-14 — llama.cpp Integration & Deployment

## Session 1: llama.cpp Integration

### What was done

- Researched llama.cpp comprehensively: MIT license, 80+ model architectures, GGUF format, OpenAI-compatible server API, Metal acceleration, speculative decoding, multimodal support, grammar-constrained generation, function calling
- Created new `LlamaCppKit` module with two files:
  - **LlamaCppProvider** — actor conforming to `InferenceProvider`, manages llama-server as a subprocess and communicates via HTTP (OpenAI-compatible API). Handles server lifecycle, health checks, streaming SSE parsing, and binary resolution. Sets DYLD_LIBRARY_PATH for dynamic library resolution.
  - **GGUFScanner** — scans known directories (LM Studio, Ollama, HuggingFace cache, Teale cache) for `.gguf` files, validates GGUF magic headers, and extracts model metadata from filenames.
- Integrated into AppCore:
  - Added `llamaCpp` case to `InferenceBackend` enum
  - Added `LlamaCppProvider` instance to `AppState`
  - Added `llamaCppBinaryPath` user setting (persisted via UserDefaults)
  - Added `scannedGGUFModels` property and GGUF scanning in `scanLocalModels()`
  - Added `loadGGUFModel()` method that switches to llama.cpp backend and loads model
  - Added GGUF auto-reload on startup when backend is llama.cpp
- Updated UI:
  - Settings: llama.cpp now appears in backend picker with binary path configuration
  - Model Browser: GGUF models appear in local models section alongside SafeTensors models
  - Added `GGUFModelRowView` with GGUF badge and source badges (LM Studio, Ollama, HF, etc.)

## Session 2: Deployment — Qwen3-32B-Q8_0 on Mac Studio

### What was done

**Mac Studio (tailor64):**
- Installed llama-server (v8792) from pre-built binary to `~/.local/bin/` with all dylibs
- Downloaded Qwen3-32B-Q8_0.gguf (34.8GB) from Qwen/Qwen3-32B-GGUF to `~/Library/Application Support/Teale/gguf/`
- Deployed latest Teale.app (167c71e) to `/Applications/Teale.app`
- Configured defaults: backend=llama_cpp, binary path=`~/.local/bin/llama-server`
- llama-server running directly on port 11435 with Metal GPU, 4 parallel slots, 8192 ctx
- Inference confirmed working: Qwen3-32B-Q8_0 responding via /v1/chat/completions

**Mac Mini (ochou / ai@100.99.93.109):**
- Deployed latest Teale.app to `/Applications/Teale.app`
- Updated Hermes config: model changed from `mlx-community/Qwen3-32B-4bit` to `Qwen3-32B-Q8_0.gguf`
- Hermes still routes to Mac Studio via Tailscale direct HTTP (`100.88.240.125:11435`)
- Inference from Mac Mini → Mac Studio confirmed working
- Hermes gateway process restarted

### Key Decisions
- **Running llama-server directly on port 11435** instead of through Teale's subprocess management, because the auto-reload of GGUF models through Teale isn't fully working yet. The direct approach is more reliable for now.
- **Mac Mini doesn't need llama-server or GGUF model** — it only has 16GB RAM, can't fit Qwen3-32B-Q8_0 (~35GB). Continues to route inference to Mac Studio.
- **No API key on llama-server** — llama-server doesn't enforce auth, but Hermes sends the old Teale API key in requests which llama-server simply ignores (returns 200 regardless).

### Issues
- **SSH to CDPO16 (100.102.161.36)** — times out, Tailscale may be disconnected
- **SSH to ochou** — SSH key rejected, had to use `ai@100.99.93.109` instead
- **SCP corrupts .app bundles** — code signing breaks during SCP transfer. Fixed by deploying via tar.gz + remote codesign.

## Session 3: Refactor — Teale Manages llama-server

### What was done

**Root cause of auto-reload failure:** The app bundle identifier is `com.teale.app` but `defaults write` was using `com.teale.InferencePool`. UserDefaults was reading from the wrong domain, so `inferenceBackend` always returned `local_mlx`.

**LlamaCppProvider refactored:**
- Default 65K context (YaRN rope scaling for >40K), batch size 4096, q8_0 KV cache
- `--reasoning off` disables thinking by default
- Runtime `updateConfiguration()` for all settings
- Health check timeout 120s (large models need time)
- Server logs to `/tmp/llama-server.log`
- Process crash detection during health check

**AppState fixes:**
- GGUF auto-reload loads directly on `llamaCppProvider` before engine wrapping (avoids provider chain timing issues)
- `loadGGUFModel` loads on provider first, then sets engine provider chain, eliminating race condition from `inferenceBackend` didSet
- `applyInferenceBackendSelection` passes binary path to llamaCpp provider

**RemoteControlBridge:**
- `/v1/app/models/load` now supports GGUF models (resolves by filename, id, or path)
- Triggers `loadGGUFModel` for GGUF files, falls through to catalog for MLX

**Deployed and verified:**
- Mac Studio: Teale manages llama-server subprocess, auto-loads Qwen3-32B-Q8_0 on startup
- Inference through Teale API (`/v1/chat/completions`) confirmed working
- Mac Mini → Mac Studio inference over Tailscale confirmed working
- Mac Mini also updated to latest Teale build

### PRs Merged
- PR #40: Refactor llama.cpp provider for production use
- PR #41: Remove debug logging

### Key Decisions
- **Bundle ID was `com.teale.app` not `com.teale.InferencePool`** — this was the root cause of all auto-reload failures. Must use correct domain for `defaults write`.
- **Direct provider loading during auto-reload** — loading on `llamaCppProvider` directly before `buildActiveInferenceProvider()` wraps it ensures the model is loaded before any provider chain setup.
